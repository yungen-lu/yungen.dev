# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A Hugo static site for `yungen.dev` using the `hugo-bearblog` theme (vendored as a git submodule at `themes/hugo-bearblog`). Output is published to `public/` (gitignored). Local Hugo runs the v0.146+ template system; the theme upstream is still on the old `_default/`/`partials/` structure and relies on Hugo's back-compat — match the theme's structure when adding any overrides here.

## Common commands

```sh
# Local dev server (includes drafts, future-dated posts)
hugo server -D -F

# Production build
hugo

# New post (uses archetypes/default.md)
hugo new content blog/<slug>.md

# After cloning, initialize the theme submodule
git submodule update --init --recursive
```

There is no test suite, linter, or package manager config — Hugo is the only build tool.

## Content flow (important)

Most content under `content/` is **not edited in this repo**. It is generated by `.github/workflows/update.yaml`, which fires on the `trigger-workflow` repository_dispatch event, clones `yungen-lu/vault`, and runs `scripts/sync_from_vault.py` (a uv-runnable Python script with PEP 723 inline metadata — single-file, no requirements.txt). The script:

- Routes by folder: `Notes/Blog/*.md` → `content/blog/<slug>/index.md`, `Notes/Blog/Reviews/*.md` → `content/reviews/<slug>/index.md`, `Notes/Blog/About.md` → `content/about/index.md`.
- Skips notes with `draft: true`.
- Strips the Obsidian `type:` frontmatter field (collides with Hugo's reserved `type` key, which under v0.146+ formally routes template lookups). Also drops frontmatter keys with `None` values so empty Obsidian fields don't leak through.
- Converts `![[image.png]]` embeds to `![](image.png)` and copies referenced files from `vault/Attachments/` (recursive search) into each post's page-bundle folder. Embed modifiers like `|400` or `|alt` are stripped. Filenames are slugified so the resulting URLs are clean.
- Preserves the hand-maintained `_index.md` stubs for the `blog/` and `reviews/` sections; wipes everything else under those plus all of `content/about/` before regenerating.

Run locally with `uv run scripts/sync_from_vault.py /path/to/vault content`.

Implications when editing:
- Do not hand-edit individual posts under `content/blog/` or `content/reviews/` — the next sync overwrites them. Source-of-truth edits belong in the vault.
- The Hugo project structure (`hugo.toml`, `layouts/`, `archetypes/`, theme) is safe to edit here.

## Site structure

Custom code is intentionally minimal so theme updates flow through cleanly:

- `hugo.toml` — site config. Bearblog-style flat URLs (`disableKinds = ["taxonomy"]` + `ignoreErrors`). Permalinks for `blog`, `reviews`, `tags` flatten to `/:slug/`. Code highlight `catppuccin-mocha`. The `[frontmatter]` block makes `.Date` resolve `publishDate` → `date` so vault notes (where `date` is export time and `publishDate` is the real date) sort correctly. The `[[menu.main]]` blocks define the nav order; About sits there (rather than in page frontmatter) because the vault's `About.md` doesn't carry a `menu:` field.
- `layouts/index.html` — the **only** custom layout file. Renders `_index.md` content, then a list of the 10 most recent posts across `blog`+`reviews`. Kept short and using only stable Hugo APIs so it survives theme updates.
- `archetypes/default.md` — template for `hugo new` (currently unused since content is synced).
- `content/_index.md` — short markdown homepage intro. Rendered by `layouts/index.html` above the recent-posts list.

Everything else (single-post layout, section listings, header/nav/footer, base CSS, SEO tags, RSS) comes from the theme.

## Adding custom CSS or `<head>` content

Don't override `layouts/partials/style.html`. The theme exposes `layouts/partials/custom_head.html` as the official extension point — add styles or `<link>` tags there. (Use the old `partials/` path, not the v0.146 `_partials/`, to match the theme's structure.)
